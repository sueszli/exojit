Abstract
========

(1) Topic: In one sentence, what's the topic?
  There is an abstraction-performance gap in ML kernel development: high-level
  frameworks enable rapid iteration, but fused kernels still require C or CUDA
  for performance.

(2) Problem: What's the key research question?
  Can we close this gap, letting ML researchers write and fuse kernels entirely
  in Python at native-code performance?

(3) Gap: Why has nobody adequately answered this yet?
  NumPy cannot fuse operations. Numba JIT-compiles Python but lacks scheduling
  primitives for tiling and memory hierarchy. Exo provides scheduling control
  but emits C, requiring an external toolchain.

(4) Approach: What's your big idea?
  A JIT pipeline (Exo -> xDSL/MLIR -> LLVM MCJIT) that compiles scheduled
  kernels to native code within Python, with ~1us dispatch. Because it JITs
  within the Python process, the write-benchmark-tune loop stays entirely in
  Python, tightening kernel development iteration cycles.

(5) Methods: How did you do the research?
  We implement common inference kernels as fused single-pass operations and
  benchmark per-kernel against existing libraries.
  We also publish an open benchmark arena for microgpt inference on CPU and
  invite the community to beat our implementation (any library, any hardware,
  SIMD permitted, no non-CPU accelerators).

(6) Impact: Why should anyone care?
  Fused softmax achieves 11.7x speedup over NumPy by eliminating intermediate
  memory traffic and per-op dispatch. The contribution is not
  just faster compute but a narrower abstraction-performance gap: ML
  researchers get native-code performance without leaving the Python workflow
  they already use. The pipeline also demonstrates xDSL/MLIR as a viable JIT
  middle-end for scheduling DSLs.
