- TODO: exojit microgpt benchmark cleanup
- Objective: Establish a minimal, focused microgpt training path with lean kernel layout, no Python codegen hacks, and a single easy-to-follow driver.
- Target: Reduce LOC, remove dead utilities, and converge on tight responsibility boundaries before adding any new features.

Top goal
- Minimize lines of code.
- Make the benchmark structure feel tight and obvious.
- Remove fake generality and hidden runtime tricks.
- Keep only abstractions that pay for themselves.

Main criticism
- `microgpt-benchmarks-game/utils/exo_kernels.py` is not really a utilities file anymore.
- It mixes reusable Exo primitives with microgpt-specific proc factories and Python `exec` codegen.
- That boundary is backwards: the so-called utility layer depends on benchmark-specific subprocs injected from the caller.
- `microgpt-benchmarks-game/bench_exojit.py` does too much at once: model kernels, tokenization, allocation, memory planning, JIT compilation, pointer packing, training loop, and validation.
- `microgpt-benchmarks-game/utils/exo_alloc.py` pays abstraction cost without containing the abstraction. Code reaches through `_buf`, `_size`, `_offset`, `_ctype`, and raw pointers everywhere.
- The current design is compensating for toolchain limitations in user code instead of expressing the model step directly.

Dead code / low-value code
- Delete `array()` from `microgpt-benchmarks-game/utils/exo_alloc.py`. It is unused.
- Delete `empty_like()` from `microgpt-benchmarks-game/utils/exo_alloc.py`. It is unused.
- Delete `fill3()` from `microgpt-benchmarks-game/utils/exo_kernels.py`. It only saves one call site and makes the utils file noisier.
- Delete fake configurability around `n_layer = 1` in `microgpt-benchmarks-game/bench_exojit.py` unless multi-layer compilation is actually implemented.
- Stop returning keepalive objects from `tokenize()` just to preserve backing storage lifetime. That is a hidden ownership hack, not a clean API.

Reusable vs benchmark-specific

Keep as reusable Exo primitives
- `fill`
- `add`
- `matmul`
- `matmul_right_t`
- `matmul_left_t`
- `relu`
- maybe `rmsnorm`
- maybe `adam`

Keep benchmark-specific, not in utils
- `softmax_xent_row`
- `embed_token`
- `embed_rms_bwd_token`
- `attn_fwd_fused`
- `attn_bwd_fused`
- `mlp_fwd_fused`
- `mlp_bwd_fused`
- `lm_head_step_*`
- any proc that exists only for this microgpt training step

Delete or move out of utils immediately
- `_make_proc`
- `make_lm_head_step_fused`
- `make_embed_rms_fwd_tokens`
- `make_embed_rms_bwd_tokens`

Structural refactor
- Stop calling the mixed file `exo_kernels.py` if it still contains benchmark logic.
- Split into either:
  - `exo_primitives.py` for real reusable kernels
  - `microgpt_kernels.py` for model-specific kernels
- Or go even tighter:
  - put all microgpt-specific Exo kernels in `bench_exojit.py`
  - keep only true primitives in utils

Tensor / allocation cleanup
- Decide whether `Tensor` is worth keeping.
- If the answer is no, replace `exo_alloc.py` with NumPy host buffers and raw pointers.
- If keeping a custom tensor type, make it honest and small:
  - one flat storage object
  - typed views
  - explicit `ptr`
  - explicit `numel`
  - explicit `view(shape, offset)`
- Stop exposing and depending on private fields everywhere.
- Add proper dtype support if the abstraction survives. Right now `int` maps to `c_int32`, but Exo `size/index` use 64-bit in the JIT path.

Memory layout cleanup
- Do not allocate individual tensors and then repack them into a flat buffer.
- Define a layout spec first.
- Allocate one flat parameter buffer.
- Create views into it.
- Initialize the views in place.
- Do the same for grads, optimizer state, and scratch if that still helps performance.

State encapsulation
- Replace dicts of string keys for params/grads/scratch with a small typed container.
- Use a `dataclass` or `namedtuple` for:
  - params
  - grads
  - scratch
  - scalar constants
  - batch/token buffers
- Avoid stringly-typed lookups like `ptrs["scratch"]["q"]`.
- Make ownership and lifetime explicit instead of hiding it in tuple packing.

Codegen cleanup
- Remove the weird Python-generating-Python path.
- Prefer direct Exo procs with tensor arguments like:
  - `input_ids: size[BLOCK_SIZE] @ DRAM`
  - `target_ids: size[BLOCK_SIZE] @ DRAM`
- Exojit already supports `size` tensors and windows.
- Use ordinary loops over `BLOCK_SIZE` inside the proc instead of generating `target0, target1, ...` signatures.
- Pass token arrays directly rather than splatting them into the function signature.

Desired simplification path
1. Remove proc factories and replace them with direct procs that take `size[BLOCK_SIZE]` token tensors.
2. Inline `fill3`.
3. Move all microgpt-specific kernels out of `utils/exo_kernels.py`.
4. Replace string-keyed state dicts for runtime scratch/ptr bundles with typed containers.
5. Replace allocate-then-pack with layout-first allocation.
6. Re-evaluate whether `Tensor` still earns its keep after the above.

Possible end state
- one file with the microgpt Exo kernels
- one very small runtime/layout helper
- one benchmark driver

What to optimize for
- Fewer files with sharper responsibilities.
- Fewer abstractions, but stronger ones.
- Fewer raw pointer plumbing sites.
- No hidden keepalive coupling.
- No Python source generation.
- Fixed-shape specialization where it meaningfully cuts code.

Rule of thumb
- If a helper is only used by this benchmark, it should live with the benchmark.
- If an abstraction leaks `_buf`, `_size`, raw pointers, and offsets everywhere, it is not actually abstracting enough.
- If the easiest explanation of a kernel requires discussing code generation machinery, the structure is wrong.
